================================================================================
RANCANGAN SOLUSI & JAWABAN CASE STUDY: AI ACTIVATION SPECIALIST
OMZETRA MANAGEMENT DASHBOARD (EXCEL/CSV TO BI REVENUE ENGINE)
================================================================================

Dokumen ini disusun sebagai panduan presentasi (script/cheatsheet 15 menit) dan 
bukti komprehensif cara berpikir bisnis serta teknis untuk posisi AI Activation Specialist.

--------------------------------------------------------------------------------
1. CARA BERPIKIR (SYSTEMS & BUSINESS THINKING)
--------------------------------------------------------------------------------
* Pergeseran Paradigma (Paradigm Shift):
  - Solusi ini tidak dirancang sekadar sebagai "Dashboard Visualisasi Data" yang 
    membaca ulang angka historis. 
  - Solusi ini dirancang sebagai "Sistem Pendukung Keputusan Strategis (Decision 
    Support System)" yang aktif mendeteksi risiko konsentrasi bisnis, anomali 
    kinerja, dan memberikan rekomendasi aksi nyata (Actionable Insights).
  
* Pendekatan Berorientasi Bisnis:
  - Manajemen tidak butuh grafik cantik jika data di dalamnya kotor (Garbage In, 
    Garbage Out). Oleh karena itu, integritas data (data validation) diletakkan 
    di pintu gerbang pertama sebelum visualisasi dibentuk.
  - Grafik harian sering kali "lurus" atau flat jika data mentah sangat sedikit. 
    Kemampuan sistem untuk mengganti dimensi kueri secara real-time (Harian, 
    Per Produk, Per Sales, Per Customer) memecahkan masalah ini dan memberikan 
    sudut pandang 360 derajat yang dibutuhkan manajemen.

--------------------------------------------------------------------------------
2. RANCANGAN SOLUSI (SOLUTION DESIGN)
--------------------------------------------------------------------------------
Solusi kami dibangun dengan arsitektur modular terpadu (Unified Architecture):
* Input Layer: Modul Drag & Drop yang mendukung unggah banyak file sekaligus 
  (Multiple Files Upload) untuk berkas Excel (.xlsx, .xls) dan CSV, serta 
  import langsung via link Google Sheets publik.
* Validation Layer (Data Integrity Engine): Komponen backend yang secara otomatis 
  memvalidasi format tanggal, membersihkan spasi/kapitalisasi (casing), menyaring 
  duplikat transaksi, dan mendeteksi sel kosong sebelum data masuk ke database.
* Storage Layer: Database relasional SQLite dengan skema bentuk normal ketiga (3NF) 
  untuk memisahkan entitas transaksi (Sales) dengan tabel Master (Customers, 
  Products, Salespeople) guna mencegah inkonsistensi data.
* Analytics Layer (Growth & Anomalies Engine): Algoritma server-side yang 
  menghitung rasio pertumbuhan periode-ke-periode (Period-over-Period Growth) 
  dan mendeteksi anomali konsentrasi bisnis.
* Output Layer: Antarmuka modern dark-mode glassmorphism dengan kueri dinamis, 
  interaksi filter global, serta kartu rekomendasi strategis AI.

--------------------------------------------------------------------------------
3. PRIORITAS FITUR (FEATURE PRIORITIZATION MATRIX)
--------------------------------------------------------------------------------
Menggunakan kerangka kerja MoSCoW untuk menunjukkan skala prioritas yang matang:

* MUST HAVE (Wajib Ada untuk MVP & Validasi Kebutuhan Dasar):
  1. Validasi Integritas Data Unggah (Deteksi data kosong, rusak, & duplikat).
  2. Penggabungan Data (Data Accumulation) antar bulan tanpa menghapus database.
  3. Visualisasi KPI Utama (Omzet, Target, Volume Qty, Frekuensi Transaksi).
  4. Grafik Tren Penjualan Multidimensi (Bisa di-switch: Waktu, Produk, Sales, Customer).

* SHOULD HAVE (Penting untuk Analisis Bisnis yang Tajam):
  1. Filter Global (Pilihan rentang waktu, sales, customer, dan produk di atas page).
  2. Growth Comparator (Persentase pertumbuhan KPI dibanding periode sebelumnya).
  3. Strategic AI Insights (Rekomendasi bisnis berformat: Temuan ➔ Dampak ➔ Solusi).
  4. Pemisahan Top Produk berdasarkan Omzet (Cash Cow) vs Kuantitas (Velocity).

* COULD HAVE (Fitur Tambahan untuk Kenyamanan Pengguna):
  1. Fitur edit target penjualan dan harga produk langsung (Inline Editor) bagi Admin.
  2. Import langsung via link Google Sheets publik tanpa unduh manual.
  3. Sistem autentikasi pengguna (Role-based access: Admin & Regular User).

* WON'T HAVE (Ditunda untuk Fase Pengembangan Berikutnya):
  1. Peramalan penjualan masa depan (Sales forecasting) berbasis Machine Learning.
  2. Integrasi API POS (Point of Sales) langsung secara real-time.

--------------------------------------------------------------------------------
4. ALUR KERJA SISTEM (AUTOMATED DATA PIPELINE)
--------------------------------------------------------------------------------
Alur data berjalan secara linier dan otomatis di dalam sistem:
[File Excel/CSV / Google Sheets] 
   ➔ (1) Diunggah via UploadModal (Multiple Upload / URL)
   ➔ (2) Diterima oleh API Upload Serverless Route
   ➔ (3) Diproses oleh Data Integrity Engine (Pemeriksaan header & validasi tipe data)
   ➔ (4) Baris cacat dilewati, duplikat disaring, dan log validasi disimpan ke 'system_metadata'
   ➔ (5) Data valid dimasukkan ke Relational Database 3NF (sales, customers, products, salespeople)
   ➔ (6) Pengguna memicu filter global di dashboard
   ➔ (7) API Dashboard memproses kueri SQL terindeks dengan filter aktif + kueri periode lalu
   ➔ (8) Algoritma AI merumuskan Strategic Insights & Recommendations
   ➔ (9) Dashboard merender visualisasi KPI, growth badges, grafik interaktif, dan rekomendasi aksi

--------------------------------------------------------------------------------
5. RISIKO YANG PERLU DIPERHATIKAN & MITIGASI
--------------------------------------------------------------------------------
* Risiko 1: Format File Unggahan Berantakan (Kualitas Data Rendah)
  - Dampak: Kueri melambat, data cacat merusak kalkulasi KPI, atau aplikasi crash.
  - Mitigasi: Validasi ketat di backend server. Sistem menyaring baris kosong, 
    memvalidasi regex tanggal, menolak format file non-spreadsheet, dan menyimpan 
    seluruh log audit ke panel khusus "Status Validasi Data" untuk transparansi pengguna.
    
* Risiko 2: Ketergantungan Kuota Target Statis
  - Dampak: Angka target penjualan dalam file Excel sering kali tidak proporsional 
    atau ketinggalan zaman, menghasilkan pencapaian bias (achievement >300%).
  - Mitigasi: Menyediakan Control Panel eksklusif Admin untuk menyunting target 
    penjualan sales dan harga produk secara dinamis langsung dari dashboard.

* Risiko 3: Bottleneck Performa pada Skala Data Besar
  - Dampak: Waktu tunggu kueri filter menjadi lambat saat transaksi mencapai jutaan.
  - Mitigasi: Membuat indeks kueri relasional (`CREATE INDEX`) pada kolom kunci 
    (`tanggal`, `customer_id`, `product_id`, `sales_id`). Struktur database 
    dirancang 3NF sehingga sangat mudah dimigrasikan ke PostgreSQL jika volume 
    data melonjak pesat.

--------------------------------------------------------------------------------
6. REKOMENDASI TEKNOLOGI & PENDEKATAN SCALABILITY
--------------------------------------------------------------------------------
* Core Framework: Next.js (React)
  - Alasan: Mendukung Server-Side Rendering (SSR) dan API Routes serverless. Ini 
    menjamin kueri database berjalan aman di sisi server (tidak terekspos ke client) 
    dan memberikan waktu muat halaman (*load time*) yang instan.
* Database: SQLite (MVP Phase) ➔ PostgreSQL (Production Phase)
  - Alasan: SQLite sangat ringan, tanpa konfigurasi rumit, dan tersimpan dalam 
    satu berkas lokal yang portabel untuk demonstrasi prototype di VPS. Namun, 
    skema relasional 3NF yang kami terapkan menjamin migrasi ke database skala 
    industri (seperti PostgreSQL) dapat dilakukan tanpa mengubah logika kode program.
* Parsing Library: SheetJS (xlsx)
  - Alasan: Standard industri tercepat untuk mengekstrak data dari biner Excel 
    (.xlsx, .xls) langsung di memori backend tanpa membutuhkan server java tambahan.
* UI & Visualisasi: Tailwind CSS + Lucide Icons + SVG-based Dynamic Graphics
  - Alasan: Menghasilkan desain bertema gelap (*neon dark-mode*) yang mewah, responsif, 
    dan grafik SVG murni yang sangat ringan serta tajam tanpa dependensi library chart 
    pihak ketiga yang berat.
